1/**
2 @file NetworkDevice.h
3 These classes abstract networking from the socket level to a
4 serialized messaging style that is more appropriate for games. The
5 performance has been tuned for sending many small messages. The
6 message protocol contains a header that prevents them from being used
7 with raw UDP/TCP (e.g. connecting to an HTTP server).
8
9 LightweightConduit and ReliableConduits have different interfaces
10 because they have different semantics. You would never want to
11 interchange them without rewriting the surrounding code.
12
13 NetworkDevice creates conduits because they need access to a global
14 log pointer and because I don't want non-reference counted conduits
15 being created.
16
17 Be careful with threads and reference counting. The reference
18 counters are not threadsafe, and are also not updated correctly if a
19 thread is explicitly killed. Since the conduits will be passed by
20 const XConduitRef& most of the time this doesn't appear as a major
21 problem. With non-blocking conduits, you should need few threads
22 anyway.
23
24 LightweightConduits preceed each message with a 4-byte host order
25 unsigned integer that is the message type. This does not appear in
26 the message serialization/deserialization.
27
28 ReliableConduits preceed each message with two 4-byte host order
29 unsigned integers. The first is the message type and the second
30 indicates the length of the rest of the data. The size does not
31 include the size of the header itself. The minimum message is 9
32 bytes:a 4-byte type, a 4-byte header equal to "1", and one byte of data.
33
34 @maintainer Morgan McGuire, http://graphics.cs.williams.edu
35 @created 2002-11-22
36 @edited 2006-11-25
37 */

203 template<typename T> static void serializeMessage
204 (uint32 t, const T& m, BinaryOutput& b) {
205
206 b.writeUInt32(t);
207
208 // Reserve space for the 4 byte size header
209 b.writeUInt32(0);
210
211 size_t L = (size_t)b.length();
212 m.serialize(b);
213 if ((size_t)b.length() == L) {
214 // No data was created by serialization.
215 // We need to send at least one byte because receive assumes that
216 // a zero length message is an error.
217 b.writeUInt8(0xFF);
218 }
219
220 uint32 len = (uint32)b.size() - 8;
221
222 // We send the length first to tell recv how much data to read.
223 // Here we abuse BinaryOutput a bit and write directly into
224 // its buffer, violating the abstraction.
225 // Note that we write to the second set of 4 bytes, which is
226 // the size field.
227 uint32* lenPtr = ((uint32*)b.getCArray()) + 1;
228 #if defined(__GNUC__)
229 *lenPtr = gcchtonl(len);
230 #else
231 *lenPtr = htonl(len);
232 #endif
233 }

268 use G3D::ReliableConduit::waitingMessageType() to detect the incoming

307 /**
308 If a message is waiting, deserializes the waiting message into
309 message and returns true, otherwise returns false. You can
310 determine the type of the message (and therefore, the class
311 of message) using G3D::ReliableConduit::waitingMessageType().
312 */
313 template<typename T> inline bool receive(T& message) {
314 if (! messageWaiting()) {
315 return false;
316 }
317
318 debugAssert(state == HOLDING);
319 // Deserialize
320 BinaryInput b((uint8*)receiveBuffer, receiveBufferUsedSize, G3D_LITTLE_ENDIAN, BinaryInput::NO_COPY);
321 message.deserialize(b);
322
323 // Don't let anyone read this message again. We leave the buffer
324 // allocated for the next caller, however.
325 receiveBufferUsedSize = 0;
326 state = NO_MESSAGE;
327 messageType = 0;
328 messageSize = 0;
329
330 // Potentially read the next message.
331 messageWaiting();
332
333 return true;
334 }
335
336 /** Removes the current message from the queue. */
337 inline void receive() {
338 if (! messageWaiting()) {
339 return;
340 }
341 receiveBufferUsedSize = 0;
342 state = NO_MESSAGE;
343 messageType = 0;
344 messageSize = 0;
345
346 // Potentially read the next message.
347 messageWaiting();
348 }

357/**
358 Provides fast but unreliable transfer of messages. On a LAN,
359 LightweightConduit will probably never drop messages but you
360 <I>might</I> get your messages out of order. On an internet
361 connection it might drop messages altogether. Messages are never
362 corrupted, however. LightweightConduit requires a little less setup
363 and overhead than ReliableConduit. ReliableConduit guarantees
364 message delivery and order but requires a persistent connection.
365
366 To set up a LightweightConduit (assuming you have already made
367 subclasses of G3D::NetMessage based on your application's
368 pcommunication protocol):
369
370[Server Side]
371<OL>
372<LI> Call LightweightConduit::create(port, true, false),
373where port is the port on which you will receive messages.
374
375<LI> Poll LightweightConduit::messageWaiting from your main loop. When
376it is true (or, equivalently, when LightweightConduit::waitingMessageType
377is non-zero) there is an incoming message.
378
379<LI> To read the incoming message, call LightweightConduit::receive with
380the appropriate class type, which mist have a deserialize method.
381LightweightConduit::waitingMessageType tells you what class is
382needed (you make up your own message constants for your program; numbers
383under 1000 are reserved for G3D's internal use).
384
385<LI> When done, simply set the G3D::LightweightConduitRef to NULL or let
386it go out of scope and the conduit cleans itself up automatically.
387</OL>
388
389[Client Side]
390<OL>
391<LI> Call G3D::LightweightConduit::create(). If you will
392broadcast to all servers on a LAN, set the third optional argument to
393true (the default is false for no broadcast). You can also set up the
394receive port as if it was a server to send and receive from a single
395LightweightConduit.
396
397<LI> To send, call G3D::LightweightConduit::send with the target address
398and a pointer to an instance of the message you want to send.

400<LI> When done, simply set the G3D::LightweightConduitRef to NULL or let

403</OL>

577/**
578 @brief Abstraction of network (socket) functionality.
579
580 An abstraction over sockets that provides a message-based network
581 infrastructure optimized for sending many small (~500 bytes) messages.
582 All functions always return immediately.
583
584 Create only one NetworkDevice per process (a WinSock restriction).
585
586 NetworkDevice is technically not thread safe. However, as long as
587 you use different conduits on different threads (or lock conduits
588 before sending), you will encounter no problems sharing the single
589 NetworkDevice across multiple threads. That is, do not invoke the same
590 Conduit's send or receive method on two threads at once.
591
592 This assumes that the underlying WinSock/BSD sockets implementation
593 is thread safe. That is not guaranteed, but in practice seems
594 to always be true (see
595 http://tangentsoft.net/wskfaq/intermediate.html#threadsafety)
596
597 <hr>
598
599 IP networks use "network byte order" (big-endian) for
600 communicating integers. "Host byte order" is the endian-ness of
601 the local machine (typically little-endian; see
602 System::endian). The C functions htonl() and ntohl() convert 32-bit
603 values between these formats. G3D only ever exposes host byte order,
